# capture-pattern

将管用的解决思路沉淀为可复用的模式。

## 执行流程

当用户调用 `/capture-pattern` 时，按以下步骤进行：

1. **识别模式来源**
   - 询问用户想要沉淀什么思路
   - 了解这个思路解决了什么问题
   - 确认这个方案已经验证有效

2. **提取核心思路**
   通过提问帮助用户梳理：
   - 遇到了什么问题？
   - 核心解决思路是什么？（3-5个步骤）
   - 为什么这个方案有效？
   - 什么情况下适用？什么情况不适用？

3. **检查相关代码**（如果有）
   - 查看 demo/ 或 src/ 中的相关实现
   - 提取关键代码片段作为示例
   - 但记住：模式是思路，不是代码

4. **生成模式文档**
   使用 `patterns/template.md` 格式：
   ```markdown
   # [模式名称]

   ## 问题
   [清晰描述问题]

   ## 解决思路
   1. 步骤1
   2. 步骤2
   3. 步骤3

   ## 适用场景
   - 场景1
   - 场景2

   ## 示例
   [具体应用示例或代码片段]

   ## 注意事项
   - 注意点1
   - 注意点2
   ```

5. **更新模式索引**
   - 在 `patterns/README.md` 中添加新模式
   - 按类别组织（开发方法论、技术方案、设计模式等）

6. **确认和保存**
   - 展示生成的模式文档
   - 询问用户是否需要修改
   - 保存到 `patterns/[pattern-name].md`

## 原则

- 模式是方法论，不是具体代码
- 关注"为什么"而不只是"怎么做"
- 保持简洁，一个模式解决一个问题
- 必须是经过验证的方案
- 适用场景要明确

## 模式分类建议

- **开发方法论** - 工作流程、开发策略
- **技术方案** - 具体技术问题的解决方案
- **设计模式** - 代码组织和架构模式
- **调试技巧** - 问题排查和调试方法
- **性能优化** - 性能相关的思路
